# AI-Forge 系统架构设计文档

**AI-Forge** 是一个生产级多租户 AI Agent 平台，基于 Anthropic Managed Agents 七组件架构设计，支持多会话并发、多智能体协同、上下文压缩、插件化工具生态，可部署在本地开发环境或 Kubernetes 集群。

## 平台架构

### 多租户隔离模型

AI-Forge 采用 **Platform → Tenant → User** 三层隔离结构：

```
Platform
  └─ Tenant                     (组织/团队，持有权限规则、资源配额、模型配置)
      ├─ Agent                   (可复用 Agent 模板：模型·提示词·工具集·MCP·Skill)
      ├─ Environment             (沙箱运行时模板：包依赖·网络策略)
      ├─ Vault                   (租户级加密凭证)
      └─ User                   (admin / member / viewer 三级角色)
          ├─ Vault              (用户级加密凭证，子密钥独立)
          ├─ MemoryStore        (自定义知识库，可见性与写入权限独立配置)
          └─ Project            (持久工作上下文：Git 仓库·参考文件·环境覆盖)
                └─ Session      (一次对话：Brain·Sandbox·Memory·CustomTools)
```

**隔离机制：** Session ID 以 `{tenantID}/{userID}:{clientID}` 命名，不同租户/用户的会话在同一存储后端内天然隔离，无需多数据库或多 Schema。租户的权限规则和资源配额每 10 秒热更新，不重启生效。

**用户角色：**

| 角色 | 权限边界 |
|------|---------|
| `admin` | 管理 Agent 模板、Environment、用户，Tenant MemoryStore 可写 |
| `member` | 创建 Project / Session / 自定义 MemoryStore，Tenant MemoryStore 只读 |
| `viewer` | 只读，Permission 引擎强制 plan 模式 |

---

### Agent 模板系统

Agent 是**租户级可复用的 Agent 配置模板**，将模型选择、系统提示词、工具权限、MCP 服务器和 Skill 封装为一个可命名、可版本化的单元。

Session 创建时通过 `agent_id` 绑定 Agent——Brain 构造时直接读取该模板，无需在每个 Session 中重复配置。若未指定 agent_id，则自动使用租户的默认 Agent（`IsDefault=true`）。

Agent 还持有 `CallableAgents` 列表，显式声明该 Agent 可以派发子任务给哪些其他 Agent，这是多智能体调用关系的静态约束（防止任意递归）。

---

### Session 的构造与绑定

Session 创建时一次性完成四件事，之后不可修改：

1. **绑定 Project**（可选）：继承 Git 仓库配置、参考文件和 Environment 覆盖；工作空间的 git clone 和文件下载异步完成，不阻塞 Session 就绪
2. **绑定 Agent**（可选）：确定本次会话使用的模型、系统提示词、工具集；未指定则使用租户默认 Agent
3. **注册 Custom Tools**（可选）：客户端将工具的完整定义（名称、描述、JSON Schema）内联传入，后端存储在 Session 记录内，不建立独立的工具注册表
4. **挂载 Memory Stores**（可选）：指定本次 Session 额外挂载的自定义知识库 ID 列表

**Environment 解析优先级**（Session 创建时确定，注入 Sandbox Pool）：
```
Project 指定的 EnvironmentID  >  租户默认 Environment  >  空环境
最后叠加 Project 内联 Env（最高优先级）
```

### MCP 与 Skill

**MCP（Model Context Protocol）** 允许为 Agent 挂载外部工具服务器：
- `stdio`：本地子进程（npx、python 等），Brain 构造时启动并管理生命周期
- `SSE/HTTP`：远程服务，支持自动重连与指数退避

工具名自动规范化为 `mcp__{server}__{name}` 格式，防止命名冲突。

**Skill** 是以 Markdown 文件定义的提示词模板。frontmatter 声明元信息（名称、描述、模型覆盖等），正文是系统提示词片段，支持 `$ARGUMENTS` 占位符在调用时展开。Skill 来源分为编译进二进制的 Bundled Skills 和运行时从数据库加载的 Dynamic Skills。

---

### Vault 凭证服务

Vault 解决 MCP 工具启动时需要注入敏感凭证（API Key、Token 等）的问题，同时保证凭证不进入 LLM 上下文、不进入工具执行容器。

加密方案：AES-256-GCM，每个用户从平台主密钥（`FORGE_VAULT_KEY`）通过 HKDF-SHA256 派生独立子密钥，不同用户的凭证密文互不可解。配置中以 `vault:secret_name` 引用，Brain 构造 MCP 客户端时解密注入进程环境，之后明文不再持有。

## Agent 架构

### 七组件栈总览

```
                    ┌──────────────────────────────────┐
   User Request ──▶ │         Orchestration            │  HTTP + JWT 认证
                    └────────────────┬─────────────────┘
                                     │
                    ┌────────────────▼─────────────────┐
                    │           EventBus               │  SSE 事件分发
                    └────────────────┬─────────────────┘
                                     │
                    ┌────────────────▼─────────────────┐
                    │             Harness              │  效果循环
                    │  HITL Gate · Custom Tool · Grader│
                    │  ┌──────────┐  ┌──────────────┐  │
                    │  │ History  │  │    Memory    │  │
                    │  └────┬─────┘  └──────┬───────┘  │
                    │  ┌────▼───────────────▼────────┐ │
                    │  │           Brain             │ │  LLM 推理（无状态）
                    │  │   ToolRegistry (声明层)      │ │
                    │  └────────────┬────────────────┘ │
                    │               │ tool_use          │
                    │  ┌────────────▼─────────────────┐ │
                    │  │           Session            │ │  Append-Only 事件日志
                    │  └──────────────────────────────┘ │
                    └────────────────┬─────────────────┘
                                     │ Execute(name, input)
                    ┌────────────────▼─────────────────┐
                    │        Sandbox + Resources       │  工具执行 + 工作空间
                    └──────────────────────────────────┘
```

**核心设计原则：**
- **单向依赖**：Orchestration → EventBus → Harness → Brain/Session/History；Sandbox 独立于 Brain，由 Eino ADK 内部调度
- **推理与执行分离**：Brain 只持有 ToolRegistry（工具的 JSON Schema），不感知工具如何运行；Sandbox 只执行，不感知推理过程
- **无状态推理**：Brain 本身不持有对话状态，状态完全由 Session（事件日志）和 History（压缩快照）承载

---

### Orchestration 层

Orchestration 是唯一感知多租户上下文的层。它的职责是：
- 解析 JWT，将 tenantID / userID / role 注入请求 context
- 根据 tenantID 加载对应的 Permission Engine
- 组装 Brain（通过 Resolver）、Sandbox Pool、Memory Stores，注入 Harness
- 维护 EventBus 订阅，将 Harness 输出的事件流转发给 SSE 客户端

Orchestration **不包含任何推理逻辑**——它只负责把正确的组件组装好，交给 Harness 驱动。

---

### EventBus 与多节点 SSE

**问题**：多节点部署时，运行 Agent 的节点和接收 SSE 的节点可能不同，事件需要跨节点传递。

**解法**：EventBus 在 Agent goroutine 和 SSE 订阅者之间插入一个事件队列，生产者和消费者完全解耦：

```
Client          Node-1 (运行 Agent)        Redis Stream        Node-2 (推送 SSE)
  │                    │                        │                     │
  │── POST /run ──────▶│                        │                     │
  │                    │ MarkRunStart()→ cursor  │                     │
  │                    │ 启动 Agent goroutine    │                     │
  │                    │── Publish(running) ────▶│                     │
  │                    │── Publish(tool_use) ───▶│                     │
  │                    │                        │                     │
  │── GET /events ──────────────────────────────────────────────────▶│
  │                    │                        │◀── Subscribe(cursor)─│
  │                    │── Publish(message) ────▶│── 回放已发事件 ──────▶│──▶ SSE
  │                    │── Publish(idle) ───────▶│── 实时推送 ──────────▶│──▶ SSE
  │                    │                        │                     │
  │── POST /events {"type":"user.interrupt"} ──────────────────────▶│
  │                    │◀─── Interrupt(sid) ─────────────────────────│
  │                    │ WatchInterrupt → 感知中断，取消 Brain 推理     │
```

`MarkRunStart` 在 Agent goroutine 启动之前调用，记录队列尾部游标——即使客户端的 SSE 连接比 Agent 启动晚，`Subscribe(cursor)` 也能从头回放本次 Run 的所有事件，不丢失。

两种后端：
- **memory**：进程内 channel，单节点开发使用
- **redis**：Redis Streams（XADD/XREAD BLOCK），多节点部署无需 sticky session，迟连接者可回放

### Harness 效果循环

Harness 是 Agent 的驱动核心，对应 Managed Agents 的"效果循环（Effect Loop）"概念。一次完整的 Run 时序如下：

```
Harness          Session         History          Brain             EventBus
   │                │               │               │                  │
   │── emit(running)──▶             │               │                  │
   │                │               │               │  Publish(running)─▶
   │                │               │               │                  │
   │── Prepare() ──────────────────▶│               │                  │
   │                │  load events  │               │                  │
   │                │◀──────────────│               │                  │
   │                │   [压缩检查，必要时触发 GlobalCompact]              │
   │◀── msgs ───────────────────────│               │                  │
   │                │               │               │                  │
   │── emit(user) ──▶               │               │                  │
   │                │               │               │                  │
   │── BuildSystemContext() ─────────────────────▶(Memory)             │
   │                │               │               │                  │
   │── Run(msgs) ───────────────────────────────────▶                  │
   │                │               │   ┌─ tool_use ─┤                  │
   │                │               │   │ Permission │  Publish(tool_use)▶
   │                │               │   │ Interceptor│                  │
   │                │               │   │ Sandbox    │                  │
   │                │               │   └─ tool_result▶ Publish(result)─▶
   │                │               │   ┌─ token chunk▶ Publish(message)▶
   │◀── reply ──────────────────────────────────────│                  │
   │                │               │               │                  │
   │── emit(assistant)▶             │               │                  │
   │                │               │               │                  │
   │── emit(idle) ──▶               │               │  Publish(idle) ──▶
```

Harness 不直接操作 HTTP，也不感知多租户。它接收已组装好的 Brain 和 Memory，只负责驱动一轮推理。工具调用事件通过 `ToolNotifCh` 异步转发，不阻塞主推理流程。

### HITL 人工确认门控

HITL（Human-In-The-Loop）让用户在工具执行前介入确认，适合需要人工审批的高风险操作。

设计要点：`HITLGate` 函数通过 context 注入，Permission Interceptor 在工具调用通过规则检查后、进入 Sandbox 之前调用它。Gate 挂起当前 goroutine，等待客户端通过独立 HTTP 请求送回结果，再恢复执行。

```
Client              Brain goroutine (Node-1)     PendingStore       SSE (任意节点)
  │                         │                        │                   │
  │                 tool_use("bash", "rm -rf /tmp")  │                   │
  │                         │                        │                   │
  │                 Permission Interceptor           │                   │
  │                   规则检查通过                     │                   │
  │                         │                        │                   │
  │                   HITLGate(toolName, args)        │                   │
  │                         │── Register(actionID) ─▶│                   │
  │                         │                        │                   │
  │                         │                        │ Publish(requires_action)▶
  │◀─────────────────────────────────────────── SSE: session.requires_action
  │                         │                        │                   │
  │  (用户看到确认框)          │  ← goroutine 挂起 →   │                   │
  │                         │                        │                   │
  │── POST /events ──────────────────────────────────────────────────────▶
  │   {type:"user.tool_confirmation", confirmed:true} │                   │
  │                         │                        │                   │
  │                         │       Deliver(actionID, {Confirmed:true})  │
  │                         │◀── resultCh ───────────│                   │
  │                         │  goroutine 恢复          │                   │
  │                         │                        │                   │
  │                   Sandbox.Execute("bash", ...)   │                   │
```

跨节点场景：`POST /events` 可以打到任意 Coordinator 节点，PendingStore（Redis）确保结果能送达持有该 goroutine 的节点。若用户拒绝（`confirmed:false`），工具调用直接返回错误，Brain 感知后可选择其他路径。

### Custom Tool 协议

Custom Tool 是**由客户端负责执行**的工具。工具定义在 Session 创建时内联传入（name + description + JSON Schema），后端存储在 Session 记录内，不建独立表。

执行协议是"请求-等待-响应"——与 HITL 机制共用同一套 PendingStore，但方向是 Agent 主动触发、等待客户端把执行结果送回来：

```
Client                Brain goroutine              PendingStore        SSE
  │                        │                            │               │
  │               tool_use("browser_screenshot", url)   │               │
  │                        │                            │               │
  │               RemoteToolExecutor.InvokableRun()     │               │
  │                        │── Register(callID) ───────▶│               │
  │                        │  goroutine 挂起             │               │
  │                        │                            │ Publish(custom_tool_use)▶
  │◀──────────────────────────────────────────── SSE: agent.custom_tool_use
  │   {tool:"browser_screenshot", tool_use_id:"...", tool_input:{url}}  │
  │                        │                            │               │
  │  (客户端执行截图)        │                            │               │
  │                        │                            │               │
  │── POST /events ──────────────────────────────────────────────────▶  │
  │   {type:"user.custom_tool_result",                  │               │
  │    tool_use_id:"...", content:"base64..."}           │               │
  │                        │                            │               │
  │                        │◀── Deliver(callID, result)─│               │
  │                        │  goroutine 恢复             │               │
  │                        │                            │               │
  │               返回截图结果给 Brain，继续推理           │               │
```

这一机制让前端能力（浏览器 DOM 操作、截图、UI 交互等）无缝参与 Agent 的工具调用循环，而无需在服务端部署对应的工具实现。一次 Brain 推理轮次可以并行触发多个 Custom Tool，每个都有独立的 `tool_use_id` 对应，客户端可按任意顺序返回结果。

### Outcome Grader

Outcome Grader 支持**目标驱动的迭代执行**：用户定义成功标准（rubric），Harness 在每轮 Agent 执行后自动评测，决定是继续迭代还是结束。

```
Client                    Harness                  Grader Brain
  │                          │                          │
  │── user.define_outcome ──▶│                          │
  │   rubric: ["单元测试全绿", "无 lint 错误"]             │
  │   max_iterations: 5      │                          │
  │                          │                          │
  │                   ┌──────▼──────────────────────┐   │
  │                   │  iter = 1                   │   │
  │                   │  Brain.Run("修复测试")        │   │
  │◀─ agent.message ──│  （agent 写代码、跑测试）      │   │
  │◀─ agent.tool_use ─│                             │   │
  │                   │  RunGraderWithBrain()        │   │
  │                   │──────────────────────────────────▶
  │                   │         临时 grader session   │   │
  │                   │         只读 Brain 主动验证结果│   │
  │                   │◀─────────────────────────────────│
  │                   │   {overall:"needs_revision",  │   │
  │                   │    criteria:[                 │   │
  │                   │      {criterion:"单元测试全绿", │   │
  │                   │       status:"needs_work",    │   │
  │                   │       feedback:"test_foo 仍失败"}]}
  │◀─ outcome_eval ───│                             │   │
  │                   │  iter = 2                   │   │
  │                   │  Brain.Run("test_foo 失败，修复")  │
  │◀─ agent.message ──│  （重新修复并验证）           │   │
  │                   │  RunGraderWithBrain()        │   │
  │                   │──────────────────────────────────▶
  │                   │◀─────────────────────────────────│
  │                   │   {overall:"satisfied", ...}  │   │
  │◀─ outcome_eval ───│                             │   │
  │◀─ session.idle ───│  stop_reason: outcome_satisfied  │
  │                   └──────────────────────────────┘   │
```

Grader 使用 `WithReadOnlyBrain` 变体（移除所有写工具），确保评测过程没有副作用；临时 grader session 在评测结束后立即销毁，不污染主 session 历史。每轮 grader 反馈（per-criterion feedback）会作为上下文注入下一轮 Brain，让 Agent 知道哪些标准还未满足。

---

### Brain LLM 推理层

Brain 是**无状态的 LLM 推理封装**。它接收历史消息列表，通过 Eino ADK Runner 驱动多轮工具调用循环，返回事件流。Brain 本身不持有对话历史（由 Harness + Session 管理），不执行工具（由 Sandbox 执行），只负责推理。

```
Brain.Run(msgs):
  Eino ADK Runner:
    → Model（含 prompt cache）
    ↓ tool_use × N（一轮可并行多个）
    Permission Interceptor（HITL 检查）
    Sandbox.Execute() × N（并行）
    ↓ tool_result × N
    → Model（继续）
    ↓ 最终文本回复 → 事件流
```

Brain 支持动态变体：
- `WithReadOnlyTools`：移除写工具，用于 Grader 评测
- `WithExtraTools`：追加工具（Custom Tools 在此注入）
- `WithBrain`：替换底层模型配置（用于子 Agent 场景）

---

### Session 持久化存储

Session 的核心不变量：**事件只追加，永不删除或覆盖**。所有对话原始事件（用户消息、助手回复、工具调用/结果、状态变更、token 用量）都持久化存储，压缩快照只是加速索引，删除快照不影响历史完整性。

Session 有完整的生命周期状态机：

```
initializing ──── init_failed
     │
     ▼
   idle ◀────── running ──── rescheduling
                                │
                           terminated
```

支持的持久化后端：`memory`（测试）、`sqlite`（本地开发）、`postgres`（生产）、`redis`（高吞吐）。所有后端通过 Driver 注册模式插拔，Session 核心包不依赖任何具体实现。

---

### History 与 Context Compression

History Manager 负责为每次 Brain 调用准备消息列表，并在上下文超出阈值时触发压缩。

**压缩触发阈值：95K token**（128K 窗口 - 20K 输出预留 - 13K 安全缓冲）。这是刻意的保守设置——随着 context 积累，模型准确率会下降（Context Rot），即使模型支持更大窗口也不应调高此值。

**三层压缩管道（成本递增）：**

```
Layer 1: MicroCompact       无 LLM 开销
  └─ 保护最近 4 轮，截断旧轮次至 8K 字符

Layer 2: StripToolResults   无 LLM 开销
  └─ 所有 tool_result 截断至 2K 字符

Layer 3: GlobalCompact      消耗 LLM token
  └─ Claude 对全量历史生成摘要，替换为压缩快照
  └─ prompt-too-long 时递归重试（最多 3 次）
  └─ 失败 → fail-open：返回未压缩，不中断会话
```

压缩后保存 Snapshot，下次 Prepare 直接从快照起点重建，避免重复压缩。

### Sandbox 与工作空间

**设计原则：声明与执行分离。** Brain 通过 ToolRegistry 告诉 LLM"有哪些工具可以调用"（JSON Schema 声明），实际执行由 Sandbox 承担。两者通过接口注入，互不感知实现细节。

三级执行环境：

| 模式 | 适用场景 | 隔离级别 |
|------|---------|---------|
| **Local** | 本地开发/调试 | 无隔离，进程内直接 exec |
| **Docker** | 生产部署 | 用户级独立容器，完整文件系统/进程隔离 |
| **Kubernetes** | 大规模多节点 | 会话级独立 Pod，PVC 持久化，NetworkPolicy 网络隔离 |

**凭证隔离：** 沙箱进程（容器/Pod）无法访问 LLM API Key 等主控凭证。凭证仅在 Brain 调用 Anthropic API 时通过函数参数传递，从不注入工具执行环境变量。

**工作空间资源（Resources）：** Session 运行期间可动态挂载两类资源：
- `FileResource`：内联内容或远程 URL，写入工作空间指定路径
- `GitResource`：自动 clone 到工作空间，支持 Token 认证

**Docker Sandbox 关键设计：**
- 容器信息持久化到数据库，节点重启或跨节点 Acquire 时可恢复，不重复创建
- Singleflight 去重：同一 Session 的并发 Acquire 请求只触发一次 Start
- 两阶段清理：5 分钟空闲停容器，10 分钟陈旧清 DB 记录

**Kubernetes Sandbox 关键设计：**
- 每个 Session 创建 ClusterIP Service（稳定 DNS）+ PVC，Pod 重建后 URL/Token 不变
- Watch API 驱动 Pod 状态变化，不依赖轮询
- 两级空闲清理：30 分钟删 Pod（保留 Service + PVC）；24 小时完全释放

### Permission 权限引擎

Permission Engine 是工具执行的最后一道防线，在工具进入 Sandbox 执行之前进行访问控制。

**规则评估策略：Last-Match-Wins。** 规则按声明顺序遍历，最后一条匹配规则决定结果，允许后续规则覆盖前面的宽泛规则。

**三种权限模式：**
- `ModeDefault`：按 allow/deny 规则执行
- `ModePlan`：只读工具允许，写入工具拒绝（适合代码审查等场景）
- `ModeViewer`：始终 plan 模式（角色级强制，不可被请求覆盖）

per-request 模式覆盖：`RunRequest.Mode` 可在单次 Run 时临时切换模式（例如进入 plan 模式预览），不影响租户全局规则。viewer 角色永远强制 plan，无法被覆盖。

HITL Gate 的注入也在 Permission Interceptor 层完成——检查通过后才询问用户，拒绝后直接阻断，不触发 HITL。

### Memory 记忆系统

记忆系统是**工具驱动的文档存储**。Agent 通过 `memory_*` 工具主动读写记忆，Harness 在每次推理前将存储索引注入系统提示词，引导 Agent 选择合适的作用域存取信息。

**存储单元：** 具名 `.md` 文件，支持 Write / Edit / Read / Search / List / Delete 操作。Write 和 Edit 通过 SHA256 校验实现乐观并发控制，防止覆盖冲突。全文搜索在 SQLite 后端使用 FTS5，在 PostgreSQL 后端使用 `tsvector` GIN 索引。

**四类作用域：**

```
user:{userID}       跨会话持久，按用户隔离
project:{projID}    跨会话共享，按项目隔离
tenant:{tenantID}   同租户全员可见（admin 可写，member 只读）
custom:{storeID}    用户自建，可配置可见性和写入权限
```

**记忆的读写流程：**

```
每次 Run 开始时                          Agent 推理中
        │                                    │
Harness: BuildSystemContext()       Brain 看到的系统提示词片段:
        │                           ┌──────────────────────────────┐
        ▼                           │ # Memory                     │
组装 SessionStores 视图:             │ ## user:alice (writable)      │
  user:alice      ← writable        │   - coding-style.md          │
  project:proj-1  ← writable        │   - shortcuts.md             │
  tenant:acme     ← readonly        │ ## project:proj-1 (writable) │
  custom:api-docs ← writable        │   - architecture.md          │
  custom:team-kb  ← readonly        │ ## custom:api-docs (writable)│
        │                           │   - endpoints.md             │
        │ 注入系统提示词               └──────────────────────────────┘
        ▼                                    │
  Brain 感知到记忆存储及其权限        Agent 决定写入:
                                    memory_write("user:alice",
                                                "shortcuts.md",
                                                "Ctrl+P = 快速打开...")
                                            │
                                    MemoryStore.Write() → SQLite/Postgres
                                            │
                                    下次 Run 时自动出现在 user:alice 的索引中
```

`MergeList()` 为 `memory_list` 工具提供跨存储的统一文件索引，Agent 可以搜索所有已挂载存储，不需要指定存储前缀。

### 多智能体协同

多智能体基于"进程内同步派发"模型——Coordinator Brain 通过 `dispatch_agent_task` 工具调用子 Agent，子 Agent 在同一进程的独立 goroutine 中运行，结果直接作为工具返回值交回 Coordinator。

**Claude 一轮可并行返回多个 tool_use**，因此多个子 Agent 可以真正并发执行：

```
用户                Coordinator Brain              Sub-Agent A        Sub-Agent B
  │                        │                           │                   │
  │── "分析这个 PR" ────────▶│                           │                   │
  │                        │  (Brain 规划，决定并行派发)  │                   │
  │                        │                           │                   │
  │             ┌──── tool_use × 2（同一轮）────────────────────────────────┐
  │             │   dispatch("explore-agent",         │                   │
  │             │           "找出所有修改的接口")         │                   │
  │             │   dispatch("verify-agent",          │                   │
  │             │           "运行相关单元测试")          │                   │
  │             └────────────────────────────────────────────────────────┘
  │                        │                           │                   │
  │             AgentTool 并发执行（各占一个信号量槽）      │                   │
  │                        │────── goroutine ──────────▶│                   │
  │                        │────── goroutine ───────────────────────────▶  │
  │                        │                           │ read/glob/grep    │
  │                        │                           │ (只读工具)          │
  │                        │                           │                   │ bash(go test)
  │                        │                           │                   │ (沙箱执行)
  │                        │◀──────────────────────────│ "找到 3 个接口变更"  │
  │                        │◀──────────────────────────────────────────────│ "测试全绿"
  │                        │                           │                   │
  │             ┌─ tool_result × 2 ──────────────────────────────────────┐
  │             │  ["找到 3 个接口变更", "测试全绿"]                         │
  │             └────────────────────────────────────────────────────────┘
  │                        │  (Brain 综合结果，生成最终回复)
  │◀── "PR 修改了 3 个接口，测试全绿，建议关注..." ──────│
```

子 Agent 拥有独立的 Session（独立事件日志）和独立的 ToolNotifCh——其工具调用事件只写入自己的子 Session，不会泄漏到父 Session 的 SSE 流中。

**并发控制：** 信号量限制最大 25 个并发子 Agent（`maxConcurrentThreads = 25`）。超出时新派发的任务会阻塞等待，不会失败。

**调用关系约束：** Coordinator Agent 必须在 `CallableAgents` 中显式列出可派发的子 Agent ID。子 Agent 若未配置自己的 `CallableAgents`，则 `dispatch_agent_task` 工具不会注册，防止无限递归。

**与传统任务队列的对比：**

```
                  旧架构（PostgreSQL 队列）        现架构（进程内同步）
  结果交付方式      pg_notify → 下一轮用户消息       工具返回值，当轮获得
  执行位置         独立 Worker 进程               同进程 goroutine
  结果延迟         多一个 LLM 推理轮次             零延迟
  状态管理         任务表 + notify 轮询            信号量 + channel
  跨节点分发        支持                          不支持
```

## 部署架构

### 单节点部署

适用于本地开发和小规模生产：

```
┌─────────────────────────────────────────────────────┐
│                   forge 单进程                        │
│                                                      │
│   HTTP :8080                                         │
│   EventBus: memory    PendingStore: memory           │
│   SubAgent: goroutines (max 25)                      │
│                                                      │
│   Storage: SQLite（sessions / memory / tasks / store）│
│   Sandbox: local (dev) / docker (prod)               │
└─────────────────────────────────────────────────────┘
```

### 多节点 Kubernetes 部署

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            Ingress / LB                                  │
└───────────────┬─────────────────────────────────────────────────────────┘
                │  （无需 sticky session）
                ▼
┌────────────────────────────────┐       ┌────────────────────────────────┐
│  forge-coordinator × N          │       │  Redis                          │
│  HTTP :8080                     │ ────▶ │  EventBus (Streams)             │
│  Brain + SubAgent(goroutines)   │       │  PendingStore (Hash + PubSub)   │
└────────────────────────────────┘       └────────────────────────────────┘
                │
                ▼
        PostgreSQL
        (sessions / memory / tasks / store)
                │
┌─────────────────────────────────────────────────────────────────────────┐
│  User Sandbox Pods（每 Session 独立）                                      │
│  ClusterIP Service（稳定 DNS）+ PVC（持久化工作空间）+ NetworkPolicy       │
└─────────────────────────────────────────────────────────────────────────┘
```

多节点部署的核心是**共享状态**：所有 Coordinator 节点通过 PostgreSQL 共享会话数据，通过 Redis EventBus 分发 SSE 事件，通过 Redis PendingStore 跨节点传递 HITL 确认和 Custom Tool 结果。Coordinator 节点本身无本地状态，可随时水平扩容。
